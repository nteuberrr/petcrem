# Plan de migración: Google Sheets → Postgres (Supabase)

Estado: **propuesta / no ejecutado**. Estrategia acordada con el cliente: construir TODO en
paralelo, dejarlo andando junto a Sheets, **probar**, y recién ahí hacer el cambio (cutover).
Si la prueba no convence, no se migra — Sheets sigue siendo la fuente de verdad. Rollback = un
flag.

Respaldo previo de Sheets: `respaldo sheets/<timestamp>/` (script `scripts/respaldo-sheets.ts`).
Al 2026-06-09: 34 hojas reales + 2 basura, ~1.523 filas.

---

## 0. Por qué migrar (resumen)
- Lo que duele NO es el almacenamiento (1.523 filas es nada), sino: **cuota de lecturas** de la
  Sheets API (ya topada en certificados), **sin transacciones/concurrencia** (`getNextId` =
  `max(id)+1` → IDs duplicados si dos procesos crean a la vez), **sin índices/queries** (se lee la
  hoja entera y se filtra en JS), **latencia** por llamada (~300ms–1s).
- Ya usamos Supabase/Postgres (mailing logs + Mensajes), así que no es tecnología nueva.

## 1. Destino
- **Reusar el proyecto Supabase de mailing** (`SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY`) para las
  tablas principales. Así seguimos con **2 proyectos** (mailing+main / Mensajes) → cabe en free
  tier. (El de Mensajes queda intacto.)
- Free tier alcanza de sobra (500 MB). Pro ($25/mes) cuando se quiera backups gestionados + sin
  pausa por inactividad.
- RLS **on, sin policies** (igual que hoy): todo el acceso es server-side con service_role.

## 2. Limpieza previa (antes de exportar el esquema)
- Borrar las pestañas basura de Sheets: `vet_convenio_eutanasia_conflict615103353`,
  `precios_eutanasia_conflict319290429` (0 filas, artefactos de sync).
- Decidir qué NO migra (queda en Sheets, casi estático): `precios_generales/convenio/especiales`,
  `especies`, `tipos_servicio`, `empresa_config`, `categorias_productos`, `_README`. (Opcional —
  se pueden migrar también; no son hot.)
- Lo que YA está en otro lado y no se toca: `mailing_logs` (Supabase mailing), cluster Mensajes
  (Supabase Mensajes). `mailing_veterinarios` / `mailing_campanas` viven en Sheets hoy → migran.

## 3. Esquema (mapa hoja → tabla) — DECISIÓN FASE 1: columnas `text`
Para que el cambio sea un **drop-in** (la app pasa todo como string y usa `''` para vacíos),
cada tabla se crea con:
- `id bigint generated by default as identity primary key` → resuelve el race de `getNextId`
  (se reemplaza por la función SQL `next_id(table)` = `nextval` atómico).
- **Todas las demás columnas `text not null default ''`** → fidelidad total con el shape actual.
- `RLS on, sin policies` (solo service_role server-side).
- **Índices** en columnas que filtramos (clientes.estado/codigo/ciclo_id/…, ciclos.fecha,
  certificados.cliente_id, cotizaciones_eutanasia.estado, solicitudes_retiro.estado, etc.).

Tipado fuerte (`date`/`timestamptz`/`numeric`/`jsonb`) y **FKs** quedan para una fase de
**hardening posterior**, ya sobre Postgres y con datos limpios — NO son necesarios para el cutover y
añaden fricción (la app pasa strings y `''`, que no entran directo en columnas tipadas).

El DDL se genera con `scripts/generar-schema-sql.ts` desde el mapa canónico
[lib/sheets-schema.ts](../lib/sheets-schema.ts) (misma fuente que `/api/init-sheets`) → archivo
`supabase/schema-principal.sql` (34 tablas; excluye `mailing_logs`, que ya vive en el proyecto
Supabase de mailing).

### 3.1 Fechas y horas (CRÍTICO)
Sheets, leído con `UNFORMATTED_VALUE`, NO devuelve ISO sino **serial de Excel**: las fechas son
números (entero = fecha; con fracción = fecha+hora) y las horas son fracción de día (0.6667 = 16:00).
La app, en cambio, escribe ISO (`YYYY-MM-DD` / `…THH:MM:SS`) y `HH:MM`. Si guardáramos el serial como
texto, la columna quedaría **mezclada** (serial viejo + ISO nuevo) → orden cronológico y filtros por
string rotos.

→ El **import normaliza todo a forma canónica** ([lib/sheets-column-types.ts](../lib/sheets-column-types.ts),
`normalizarCelda(col, raw)`): fecha→`YYYY-MM-DD`, fecha+hora→`YYYY-MM-DDTHH:MM:SS` (local, sin shift
UTC), hora→`HH:MM`, booleano→`TRUE`/`FALSE`. La clasificación es **explícita por nombre** (columnas
`fecha*`/`vigente_desde`/`enviado_ultima_fecha` = fecha; set `COLUMNAS_HORA` = hora; set
`COLUMNAS_BOOL` = boolean) — NUNCA por "es un número", para no convertir por error `peso`/`lat`/`lng`/
`precio`/`monto`/`litros` (que también caen en rango de serial). Resultado: ISO ordena
lexicográficamente = cronológicamente, y `lib/dates` (que ya entiende ISO y serial) sigue mostrando
todo igual. De aquí en adelante la app escribe ISO → columna 100% consistente.

## 4. Capa de datos con doble backend (el corazón del plan)
Crear `lib/datastore.ts` que exponga **las mismas firmas** que hoy usan las rutas, con un flag de
backend:

```
DATA_BACKEND = 'sheets' | 'postgres'   // env, default 'sheets'
```

- `getSheetData(name)` → en postgres: `SELECT * ORDER BY id` devolviendo `Record<string,string>[]`
  (mismo shape; se castea a string para compatibilidad). Las rutas que leen+filtran en JS siguen
  funcionando sin tocarlas.
- `appendRow(name, obj)` → `INSERT ... RETURNING id` (id por identity; se ignora el id entrante).
- `updateRow(name, rowIndex, obj)` → resuelve el `id` por la posición en el mismo `ORDER BY id`
  (compatibilidad temporal con el patrón actual `findIndex`+`updateRow`). **Deuda técnica**: migrar
  gradualmente las rutas calientes a `updateById(name, id, obj)` (lo agregamos al datastore) para
  eliminar el TOCTOU.
- `getNextId` → deja de existir / no-op (lo da la DB).
- `ensureSheet`/`ensureColumns` → no-op en postgres (el esquema lo maneja el SQL/migraciones).

`lib/google-sheets.ts` se mantiene como backend 'sheets'. `datastore` decide a quién llamar. Las
rutas pasan a importar de `datastore` en vez de `google-sheets` (cambio mecánico de import; mismas
funciones). Así, **flip del flag = cambia toda la app** sin reescribir lógica.

## 5. Import (Sheets/respaldo → Postgres)
Script `scripts/importar-a-postgres.ts` (LISTO):
- Fuente: el respaldo JSON (`respaldo sheets/<ts>/_TODO.json`) o Sheets en vivo.
- Por celda aplica `normalizarCelda(col, raw)` de [lib/sheets-column-types.ts](../lib/sheets-column-types.ts)
  → fechas/horas/booleanos a forma canónica (ver 3.1); el resto, string tal cual.
- Inserta **preservando los ids actuales** (insert con id explícito; columna identity `by default` lo
  acepta) y al final `setval` del sequence a `max(id)` para que `next_id` no choque.
- Idempotente: `truncate` + reinsert por tabla (carga inicial, no incremental).

## 6. Verificación (la "prueba" antes de migrar)
1. `scripts/verificar-migracion.ts`: compara **conteos por tabla** Sheets vs Postgres y hace
   **spot-checks** (ej. 10 fichas al azar: todos los campos iguales).
2. **Modo sombra (shadow)**: con `DATA_BACKEND=postgres` en un deploy de *preview* de Vercel (no
   producción), recorrer las pantallas clave (clientes, operaciones, certificados, eutanasias,
   correos, dashboard, reportes) y confirmar que todo se ve y funciona igual.
3. Probar **escrituras**: crear ficha, generar/enviar certificado, agendar (bot), subir foto/video.
4. Checklist de aceptación firmado → recién ahí cutover.

## 7. Cutover y rollback
- **Cutover**: re-import final (para traer lo último de Sheets), luego setear
  `DATA_BACKEND=postgres` en producción (Vercel env) y redeploy. Opcional: dejar un job que
  **espeje** Postgres→Sheets (reusando/adaptando `sync-database`) para conservar la "vista en
  planilla" y un backup vivo.
- **Rollback**: `DATA_BACKEND=sheets` + redeploy. Como Sheets quedó intacto durante toda la prueba,
  volver es instantáneo y sin pérdida.

## 8. Orden de ejecución (incremental, por dolor)
1. Crear esquema en Supabase (mailing project) + `datastore` con doble backend (default sheets → no
   cambia nada en prod).
2. Import + verificación de conteos.
3. Migrar imports de las rutas a `datastore` (mecánico).
4. Deploy preview con `DATA_BACKEND=postgres` → prueba completa (fase 6).
5. Si pasa: cutover (fase 7). Si no: se queda en sheets, sin costo de vuelta.

## 9. Costos
- Build de la migración: esfuerzo de desarrollo (capa datastore + SQL + import + verificación +
  swap de imports). Acotado porque el acceso ya está centralizado.
- Infra: **$0** (free) al inicio; **$25/mes** (Supabase Pro) cuando se quiera backups + sin pausa.

## 10. Qué ganamos
Se acaba el throttling de lecturas; IDs y concurrencia seguros; queries reales (dashboard/reportes
más rápidos y simples); base sólida para sumar features. Sheets puede quedar como espejo/backup.

## 11. Backups (paridad o mejor que hoy)
Hoy el respaldo es el Apps Script que copia la planilla a Drive cada 48 h
(`scripts/apps_script_backup.gs`) — atado a Sheets, deja de aplicar. Reemplazo con 3 capas:

1. **Dump propio automático (equivalente directo al de hoy).** `scripts/respaldo-postgres.ts`
   vuelca todas las tablas a JSON (mismo patrón que `respaldo-sheets.ts`) y lo sube a **R2** (o
   Drive), vía **cron** (Vercel Cron o GitHub Action) cada 24–48 h. Independiente del plan de
   Supabase → control e historial propios. **Obligatorio en free tier** (que no trae backups).
2. **Backups gestionados de Supabase.** Free: no hay. **Pro ($25/mes): diarios automáticos, 7 días
   de retención** (+ PITR en planes mayores). Se suma al pasar a Pro.
3. **Espejo Postgres → Sheets (opcional, recomendado).** Job que copia a la misma planilla
   (reusando `sync-database`): conserva la "vista en planilla" para mirar a mano Y mantiene vivo el
   Apps Script de Drive actual como copia secundaria. No se pierde nada de lo de hoy.
   - **Dos cadencias distintas:** (a) el **espejo** Postgres→planilla corre **cada 24 h**
     (decisión del cliente) vía cron → la planilla queda al día. (b) el **Apps Script** que copia la
     planilla a Drive sigue **cada 48 h** (sin cambios). Como el espejo (24 h) refresca más seguido
     que el snapshot de Drive (48 h), la copia en Drive siempre toma datos de ≤24 h de antigüedad.

Neto: hoy = 1 copia (Drive/48 h); post-migración = dump propio por cron + (Pro) diarios de Supabase
+ (opcional) espejo a Sheets.
